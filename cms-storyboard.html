회원 관리

01

회원 목록

Member List
회원 관리
목적
전체 회원을 조회·관리하는 화면. PII(개인식별정보)를 포함하므로 보안 1순위.
주요 동작 · 플로우
  • 상단 필터(연동상태 / 계정상태 / 마케팅 수신)·닉네임 검색으로 목록 좁히기
  • 행의 '상세'로 회원 상세(보유 자산·활동 이력·정지/해제) 진입
  • 컬럼: 닉네임/식별자 · 연동 상태 · 가입일 · 랜덤/등급/이벤트 티켓 · 마케팅 수신 · 계정 상태
상태 전환
  • 계정 상태: 정상 ↔ 정지 ↔ 탈퇴
  • 연동 상태: 게스트(미연동) → 카카오 연동 (단방향, 연동 후 사용 가능)
권한
슈퍼어드민·운영자 = 조회+관리(정지 등), CS = 조회 전용.
데이터 흐름
users 테이블(PII). 집계: 티켓 보유량·가입일·마지막 접속.
예외 · 엣지케이스 / 확정 사양
  • 애플·구글 로그인 폐지 → 가입유형이 아니라 '연동 상태(게스트/카카오)' 2축으로 필터·표시
  • 게스트는 카카오ID가 없으므로 '닉네임 + (미연동) 배지 + 내부 식별자(app_uid)'로 표시, 적립만 누적·사용 불가. 연동 시 카카오ID로 자동 대체

티켓 내역

Ticket History
티켓 · 보상
목적
티켓 발행·소모·회수를 기록하는 원장(append-only). 회계·세무·부정거래 추적의 근거.
주요 동작 · 플로우
  • 발행유형·상태 필터로 조회 → 행별 사유·티켓·상태·일시
  • '수동 지급/회수'로 관리자 직접 조정(로그 남김)
상태 전환
  • 티켓 상태: 보유 중 / 가지급(대기) / 사용 완료 / 회수
  • '사유'가 발행 종류(구매확정·가지급·출석·광고·친구초대·주간미션·관리자지급·소모·회수)를 표현
권한
슈퍼어드민·운영자.
데이터 흐름
tickets 원장. 절대 삭제 금지 — 회수 시에도 음수 레코드를 추가하고 원본 보존.
예외 · 엣지케이스 / 확정 사양
  • '발행 종류' 별도 컬럼은 '사유'와 중복이라 만들지 않음 — 사유 유지 + '상태'만 컬럼으로 승격(필터+컬럼 병행)
03

추첨 관리

Draw Management
티켓 · 보상
목적
이벤트 티켓 기반 경품 추첨 회차를 등록·운영하고 당첨자를 확정하는 화면.
주요 동작 · 플로우
  • 회차 목록(경품 구성·기간·마감·참여 티켓·당첨자·상태)
  • 상단 '+ 회차 등록'(오버레이 폼: 등수별 경품 구성) → 등록
  • 마감된 회차의 '당첨 처리(추첨 실행)' → 모달: 응모 풀 확정 → 추첨 알고리즘 → 등수별 당첨자 산출 → 발표/지급
상태 전환
  • 회차: 진행 중 → 마감·처리 대기 → 완료
  • 좌측 메뉴 (1) 배지 = '당첨 처리 대기 회차 수'. 처리 완료 시 감소 → 0이면 숨김
권한
슈퍼어드민·운영자.
데이터 흐름
draws / draw_prizes / entries(응모) 테이블.
예외 · 엣지케이스 / 확정 사양
  • 추첨 방식 = '자동(매주 월요일 자동 추첨) / 수동(관리자 실행)' 선택값
  • 등수별 경품: 같은 등수 중복 행 금지(등수당 1행), 한 등수에 경품 여러 개는 허용
  • 등록 폼은 목록과 분리(오버레이/우측 패널) + 상단 고정 버튼

예외 · 엣지케이스 / 확정 사양

  • '경품 변경 이력' 로그(버전·변경자·전후·공표 여부) 필요 — 정산·민원 감사 추적

주간 이벤트 추첨

Weekly Event Draw
티켓 · 보상
목적
이벤트 티켓 '소비처' — 보유자 자동 응모 후 추첨해 경품 지급(티켓을 새로 발급하지 않음).
주요 동작 · 플로우
  • 매주 월요일 0시 자동 응모(이벤트 티켓 보유 회원) → 당첨 확률로 추첨 → 당첨자 경품 지급
  • '수정'(Edit) 모달로 운영 파라미터 조정
상태 전환
  • 이벤트 티켓: 당회 미응모 시 1회 이월 → 최대 2회 추첨 후 소멸(소비)
권한
슈퍼어드민·운영자.
데이터 흐름
weekly_draws / event_tickets 테이블, 스케줄러(주간 추첨 집계) 연동.
예외 · 엣지케이스 / 확정 사양
  • Edit는 당첨 확률(%)·추첨 일정(요일/시각)·당첨 보상·이벤트 티켓 이월/만료 회차를 편집
  • 상시 반복 정책 1개라 회차별 '이벤트 생성'은 불필요(특별 회차는 추첨 관리 활용)
06

환수 이력

Clawback History
티켓 · 보상
목적
티켓 회수·소멸 결과와, 즉시 회수 불가한 '부채'를 추적하는 화면(두 섹션).
주요 동작 · 플로우
  • ① 티켓 회수·소멸 이력 = 처리된 회수/소멸 결과 로그(처리일시·닉네임·주문ID·환수 티켓·원인·원 적립일) — '상세'로 근거 확인
  • ② 환수 부채 추적 = 사용된 티켓에 취소·반품 발생 시 음수 잔액(부채) 기록 → 다음 적립 시 자동 상계 대기
상태 전환
  • 원인: 만료 / 반품 / 취소
  • 부채: 상계 대기 → 다음 적립 시 자동 상계
권한
슈퍼어드민·운영자.
데이터 흐름
clawbacks / clawback_debts 테이블, orders(반품·취소) 연동.
예외 · 엣지케이스 / 확정 사양
  • 상세 모달 = 원본 티켓(등급·적립일·원 주문) + 회수 사유 상세(CS 메모) + 회수 전후 잔액 + 관련 부채 연결

Point Policy
포인트 관리
목적

주요 동작 · 플로우
  • '포인트 ↔ 티켓 교환 비율'·'적립·만료 정책' 표시 → '수정' 모달로 편집
상태 전환

권한
슈퍼어드민·운영자(비용 직결, 변경 이력 필수).
데이터 흐름
point_policy(버전) / point_transactions 테이블.
예외 · 엣지케이스 / 확정 사양
  • 수정 모달 = 교환 비율(10P=1원)·만료 기간(적립일+1년·FIFO)·일일 적립 한도·교환 방향 편집 + 변경 전후 비교·이력 기록
  • '적립·만료 정책'은 필요 — 비용·민원 핵심값이라 운영자 통제·감사 가능해야 함
